# CRM-сервис рассылок (Medinternet MAX)

Отдельный сервис для сегментированных рассылок пользователям MAX-бота. Специалист
входит по логину/паролю в веб-панель, выбирает получателей, пишет сообщение и
отправляет. Рассылка идёт **токеном основного MAX-бота** с троттлингом.

Порт CRM-панели Telegram-версии. Отличия: отправка через **MAX Bot API**, поиск и
карточка пользователя — по данным MAX (**MAX ID**), очередь рассылки **внутри процесса**
(без Redis) — на Railway это один сервис.

## Архитектура

```
Веб-панель ──HTTP──> crm-api (aiohttp) ──в память──> внутренний воркер (asyncio)
                          │                                    │
                          ├── читает public.users              └── шлёт сообщения токеном
                          └── пишет статусы в схему crm             MAX-бота, пишет статусы в crm
```

- **crm-api** — REST API + веб-панель + воркер рассылки в одном процессе.
- **Neon** — та же БД, что у бота. `users` читается, служебные данные пишутся в
  отдельную схему `crm` (таблицы бота на запись не трогаем).
- Очередь — `asyncio.Queue` в памяти, один воркер + пауза = темп ≤ `SEND_RATE_PER_SEC`.
  Redis не нужен. Минус: рассылка «в полёте» не переживает перезапуск сервиса.

## Подготовка (один раз)

### 1. Схема в Neon
Выполни в SQL-редакторе Neon `migrations/001_crm_schema.sql` — создаёт схему `crm`
и таблицы `blocked_users`, `broadcast_log`. Файл `002_crm_role.sql` (отдельная
read-only роль) — по желанию; можно просто использовать тот же `DATABASE_URL`, что у бота.

### 2. `.env`
Скопируй `.env.example` в `.env` и заполни:

| Переменная          | Значение                                        |
|---------------------|-------------------------------------------------|
| `CRM_DB_URL`        | Строка Neon (та же, что у бота, либо `crm_reader`) |
| `MAX_BOT_TOKEN`     | Токен MAX-бота (тот же, что у бота)              |
| `JWT_SECRET`        | Длинная случайная строка                        |
| `CRM_LOGIN_EMAIL`   | Email специалиста для входа                      |
| `CRM_LOGIN_PASSWORD`| Пароль специалиста                              |

## Локальный запуск

```bash
docker-compose up --build      # один контейнер crm-api
# или без докера:
pip install -r requirements.txt && python -m app.main
```

Панель: **http://localhost:8090**

## Деплой на Railway

**Один сервис** из подпапки `crm-service/` (Redis не нужен):

1. New → Deploy from GitHub repo → `Medinternet_MAX`, **Root Directory = `crm-service`**.
2. Variables: `CRM_DB_URL`, `MAX_BOT_TOKEN`, `JWT_SECRET`, `CRM_LOGIN_EMAIL`,
   `CRM_LOGIN_PASSWORD` (при 401/404 от API добавь `MAX_API_BASE=https://platform-api2.max.ru`).
3. Settings → Networking → Generate Domain — это адрес панели.

## Фильтры сегмента

| Фильтр                        | Поле в `users` | Статус   |
|-------------------------------|----------------|----------|
| `created_from` / `created_to` | `created_at`   | работает |
| `has_email` (true/false)      | `email`        | работает |
| `has_phone` (true/false)      | `phone`        | работает |
| `med_ids` / `tg_ids`          | `med_id` / `telegram_id` (MAX ID) | работает |

Заблокировавшие бота (`crm.blocked_users`) автоматически исключаются из выборки.

## Безопасность

- Секреты только в `.env` (в `.gitignore`), не в коде.
- Все `/api/*` (кроме логина) — под JWT.
- `preview` возвращает только количество — защита от случайной рассылки на всю базу.

## ⚠️ Требует проверки на боевом токене

Отправка через MAX (`app/bot_client.py`, `app/max_client.py`): базовый URL API,
схема авторизации, загрузка медиа (`/uploads`) и трактовка ошибки блокировки (403).
Текстовые рассылки — основной путь; медиа при необходимости подстроим по логам.
